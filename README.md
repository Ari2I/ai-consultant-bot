# ИИ-консультант для малого бизнеса — Telegram-бот с RAG

Telegram-бот на Python, который отвечает на вопросы клиентов малого
бизнеса (магазин, сервис, студия) на основе реальных документов
компании — FAQ, прайс-лист, условия доставки и т.п. — вместо того,
чтобы выдумывать ответы. Если бот не уверен в ответе, вопрос
передаётся владельцу бизнеса или менеджеру.

Архитектура — классический RAG (Retrieval-Augmented Generation),
реализованный без LangChain: поиск релевантных фрагментов документов
по косинусному сходству эмбеддингов → сборка промпта → генерация
ответа через GigaChat API. Каждый шаг пайплайна — это простой,
самостоятельно написанный и протестированный модуль, а не готовая
"чёрная коробка".

## Возможности

- **Ответы строго по документам компании.** Системный промпт прямо
  запрещает модели придумывать факты; если релевантность найденного
  контекста ниже настраиваемого порога — вопрос не отправляется в
  LLM вообще, а сразу эскалируется человеку.
- **Управление базой знаний через Telegram-команды**, без веб-панели:
  `/kb_add` (загрузка PDF/DOCX/XLSX/TXT/MD), `/kb_list`, `/kb_remove`.
- **Простая аналитика** (`/stats`): сколько вопросов задано всего,
  сколько эскалировано человеку, список последних эскалированных
  вопросов.
- **Локальные эмбеддинги** (sentence-transformers, CPU, офлайн) —
  единственный платный внешний вызов в проекте — это сама генерация
  ответа через GigaChat; поиск релевантного контекста ничего не
  стоит и не требует сети.
- **Однотенантная модель**: один экземпляр бота = один бизнес.
  Админы (владелец + менеджеры) управляют базой знаний, все
  остальные пользователи — клиенты, просто задающие вопросы.
- **Устойчивость к сбоям GigaChat API**: ретраи с backoff при
  временных ошибках (лимит запросов, HTTP 5xx); при окончательном
  сбое после исчерпания повторов вопрос эскалируется человеку, а не
  остаётся без ответа.
- **Rate limiting** на клиента — защита от флуда вопросами.
- Демо-база знаний вымышленного магазина одежды «ТриНитки» —
  готовый пример для демонстрации (`demo_data/`).

## Источники данных и стоимость

- **Генерация ответа**: [GigaChat API](https://developers.sber.ru/portal/products/gigachat-api)
  (Сбер). Модель по умолчанию — `GigaChat-2` (актуальна на июль
  2026 — модели первого поколения GigaChat/GigaChat-Pro/GigaChat-Max
  как отдельные больше не существуют, запросы к ним автоматически
  перенаправляются на GigaChat-2/-Pro/-Max). Авторизация — OAuth2 по
  Authorization key (`GIGACHAT_CREDENTIALS`) и scope
  (`GIGACHAT_SCOPE`), а не простой API-ключ — см. раздел
  «Авторизация и сертификат» ниже. Актуальные тарифы уточняйте в
  личном кабинете https://developers.sber.ru/studio/ перед стартом —
  они могут измениться после написания этого README.
- **Эмбеддинги для поиска**: локальная модель
  `paraphrase-multilingual-MiniLM-L12-v2` (sentence-transformers),
  работает на CPU, без API-вызовов и без затрат. Поддерживает
  русский язык. Не связана с GigaChat — эмбеддинги остаются
  локальными независимо от провайдера генерации ответа.

## Установка

Требуется Python 3.10 или новее.

```bash
pip install -r requirements.txt
```

```bash
cp .env.example .env
```

Заполните в `.env`:

- `BOT_TOKEN` — токен от [@BotFather](https://t.me/BotFather).
- `ADMIN_CHAT_IDS` — chat_id владельца бизнеса (и менеджеров), через
  запятую. Узнать свой chat_id можно у
  [@userinfobot](https://t.me/userinfobot).
- `GIGACHAT_CREDENTIALS` — Authorization key из личного кабинета
  https://developers.sber.ru/studio/ (см. раздел «Авторизация и
  сертификат» ниже).

Остальные параметры (порог уверенности, размер чанков, top-K
фрагментов, лимит запросов в минуту) можно оставить по умолчанию —
см. таблицу ниже.

## Авторизация и сертификат GigaChat

GigaChat устроен иначе, чем большинство LLM API, и здесь легко
споткнуться на старте:

1. **Авторизация двухступенчатая (OAuth2).** В `.env` указывается не
   готовый API-ключ, а `GIGACHAT_CREDENTIALS` (Authorization key) и
   `GIGACHAT_SCOPE`. SDK `gigachat` сам обменивает их на access-токен
   и обновляет его автоматически (токен живёт около 30 минут) —
   вручную ничего обновлять не нужно.
2. **Обязателен корневой сертификат НУЦ Минцифры.** GigaChat API
   использует TLS-сертификат, подписанный российским удостоверяющим
   центром, которого нет в стандартных доверенных корневых
   сертификатах операционных систем/браузеров. Без этого сертификата
   запросы будут падать с ошибкой проверки TLS-сертификата.
   Правильное решение — установить сертификат в доверенные корневые
   сертификаты системы (см. https://www.gosuslugi.ru/crt — раздел
   «Сертификат НУЦ Минцифры») или указать путь к файлу сертификата в
   `GIGACHAT_CA_BUNDLE_FILE`.
   Быстрый обходной путь для локальной разработки —
   `GIGACHAT_VERIFY_SSL_CERTS=false` в `.env`. Это отключает проверку
   TLS-сертификата целиком, поэтому **не подходит для продакшена** —
   используйте только чтобы быстро проверить, что бот вообще
   отвечает, и замените на нормальную установку сертификата перед
   реальным развёртыванием.

## Запуск

```bash
python main.py
```

При первом запуске sentence-transformers скачает файлы модели
эмбеддингов (~470 МБ) из HuggingFace Hub — нужен доступ в интернет к
`huggingface.co`. При последующих запусках модель берётся из
локального кэша.

## Демо-документы

В `demo_data/` — готовый пример базы знаний для вымышленного
магазина одежды «ТриНитки»: `about.txt`, `faq.docx`,
`price_list.xlsx`, `shipping_returns.pdf`. Загрузите их по очереди
через `/kb_add` (от имени админа), чтобы сразу проверить, как бот
отвечает на реальные вопросы. Пересоздать эти файлы можно скриптом:

```bash
pip install reportlab  # только для генерации демо-PDF, не входит в requirements.txt
python demo_data/generate_demo_documents.py
```

## Использование в Telegram

### Команды администратора (chat_id из ADMIN_CHAT_IDS)

| Команда | Что делает |
|---|---|
| `/kb_add` | Загрузить документ в базу знаний (пришлите файлом PDF/DOCX/XLSX/TXT/MD) |
| `/kb_list` | Список загруженных документов с числом фрагментов |
| `/kb_remove ID` | Удалить документ (и все его фрагменты) из базы знаний |
| `/stats` | Всего вопросов, сколько эскалировано, последние эскалированные вопросы |
| `/cancel` | Отменить текущий диалог добавления документа |

### Для клиентов

Просто напишите вопрос обычным текстом. Бот:

1. Считает эмбеддинг вопроса.
2. Ищет наиболее похожие фрагменты среди загруженных документов
   (`TOP_K_CHUNKS`, по умолчанию 5).
3. Если лучшее сходство ≥ `SIMILARITY_THRESHOLD` (по умолчанию 0.75)
   — собирает промпт из найденного контекста и вопроса, вызывает
   GigaChat, отдаёт ответ.
4. Если сходство ниже порога — сообщает, что передаёт вопрос
   менеджеру, и пересылает вопрос со сведениями о клиенте (имя,
   username, chat_id) всем админам.

`/start` и `/help` доступны всем и показывают краткую справку.

## Конфигурация

Полный список — в `.env.example`. Ключевые параметры:

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `SIMILARITY_THRESHOLD` | Порог косинусного сходства для эскалации | `0.75` |
| `TOP_K_CHUNKS` | Сколько фрагментов подавать в контекст модели | `5` |
| `CHUNK_SIZE` / `CHUNK_OVERLAP` | Параметры чанкинга документов (в символах) | `500` / `50` |
| `RATE_LIMIT_PER_MINUTE` | Лимит вопросов в минуту на клиента | `10` |
| `GIGACHAT_MODEL` | Модель GigaChat для генерации ответов | `GigaChat-2` |
| `GIGACHAT_SCOPE` | Тип доступа (физлицо/юрлицо) | `GIGACHAT_API_PERS` |
| `GIGACHAT_VERIFY_SSL_CERTS` | Проверка TLS-сертификата НУЦ Минцифры | `true` |
| `GIGACHAT_MAX_RETRIES` / `GIGACHAT_RETRY_BACKOFF_SECONDS` | Устойчивость к временным сбоям API | `2` / `1.0` |
| `EMBEDDING_MODEL_NAME` | Модель локальных эмбеддингов | `paraphrase-multilingual-MiniLM-L12-v2` |

`SIMILARITY_THRESHOLD` — самый важный параметр для подбора под
конкретный бизнес: слишком высокий порог приведёт к частым
эскалациям даже по простым вопросам, слишком низкий — к ответам на
основе слабо релевантного контекста. Рекомендуется подбирать его
методом проб на реальных вопросах клиентов после первой загрузки
документов.

## Блокировка Telegram Bot API в России

Провайдеры и системы ТСПУ в России периодически блокируют доступ
именно к `api.telegram.org` — при этом сам хост может отвечать на
`ping`, но HTTPS-соединение на порт 443 зависает по таймауту или
падает с ошибкой вида `Cannot connect to host api.telegram.org`
(на Windows — `[WinError 121] Превышен таймаут семафора`). Если бот
запускается без ошибок в остальном (модель эмбеддингов загрузилась),
но падает именно на этом этапе — это тот самый случай, а не ошибка
в коде.

**Решение — прокси.** В `.env` есть переменная `PROXY_URL` для HTTP
или SOCKS5-прокси:

```
PROXY_URL=socks5://user:password@1.2.3.4:1080
```

Если переменная задана, бот автоматически подключается к Telegram
через указанный прокси (пакет `aiohttp-socks`, уже в
`requirements.txt`). Если пустая — прямое подключение. Важно: этот
прокси влияет только на подключение к Telegram, а не на доступ к
GigaChat API или HuggingFace Hub — это независимые внешние сервисы,
и оба доступны из России напрямую без прокси.

## Архитектурные ограничения (осознанные)

- **Векторное хранилище — SQLite + numpy, без отдельной векторной
  БД.** Для масштаба "документы одного небольшого бизнеса" (десятки-
  сотни фрагментов) этого достаточно: поиск делается в памяти по
  всем векторам сразу. Для существенно большего объёма документов
  (тысячи-десятки тысяч) потребовалась бы полноценная векторная БД
  (Qdrant/FAISS/Milvus).
- **Однотенантная модель**: один бот = один бизнес. Для агентства,
  обслуживающего много клиентов одновременно, потребовалась бы
  многотенантная архитектура — отдельная, более сложная задача.
- **Без LangChain и подобных фреймворков** — пайплайн написан вручную
  ради прозрачности и простоты отладки.
- **Без голосовых сообщений, веб-панели администратора, function
  calling / агентных сценариев и дообучения моделей** — см. план
  проекта, это сознательно не входит в первую версию.

## Известные ограничения

- Качество ответа зависит от того, насколько хорошо документ разбит
  на фрагменты и насколько релевантный контекст находит поиск —
  плохо структурированный исходный документ (например, огромная
  простыня текста без абзацев) может потребовать ручной правки перед
  загрузкой.
- Сравнение "было/стало" между документами не ведётся: повторная
  загрузка обновлённого прайс-листа не заменяет старый автоматически
  — старый документ нужно удалить через `/kb_remove` вручную, иначе
  оба будут участвовать в поиске одновременно.
- Аналитика `/stats` — простая (общие счётчики и список последних
  эскалированных вопросов), без группировки похожих вопросов по
  темам — в плане это было заявлено как опциональное расширение при
  наличии времени.
- Rate limiting хранится в памяти процесса — при перезапуске бота
  лимиты обнуляются; это приемлемо для защиты от флуда, но не
  подходит как строгий биллинговый механизм.
