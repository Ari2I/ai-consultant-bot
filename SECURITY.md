# Безопасность

Краткий чек-лист того, что уже сделано в проекте и на что обратить
внимание при развёртывании. Не претендует на исчерпывающий аудит —
это практический минимум для бота такого масштаба.

## Уже реализовано в коде

- **Токен бота и credentials GigaChat API не хранятся в коде** —
  только в `.env`, который исключён из репозитория (`.gitignore`) и
  из Docker-образа (`.dockerignore`). Сертификат НУЦ Минцифры
  (`certs/`) исключён из репозитория по тому же принципу — секреты и
  доверенные материалы передаются только во время запуска контейнера
  через `env_file` и volume, в образ не попадают.
- **Разграничение доступа админ/клиент**: команды управления базой
  знаний (`/kb_add`, `/kb_list`, `/kb_remove`, `/stats`) доступны
  только chat_id из `ADMIN_CHAT_IDS` — проверка `_is_admin()`
  выполняется в каждом обработчике (см. `bot/admin_handlers.py`).
  Обычные клиенты не могут добавлять или удалять документы базы
  знаний, только задавать вопросы.
- **Загружать документы в базу знаний может только админ.** Это
  важно не только с точки зрения контента: файлы Office (DOCX/XLSX)
  технически являются ZIP-архивами с XML внутри, и разбор
  специально сконструированного вредоносного файла в принципе может
  быть источником проблем (например, "zip bomb" — файл, который при
  распаковке разрастается до огромного размера). Ограничение
  загрузки только доверенными админами существенно снижает этот
  риск по сравнению с приёмом файлов от анонимных клиентов.
- **Rate limiting на клиента** (`bot/rate_limiter.py`) — ограничивает
  число вопросов в минуту на один chat_id, чтобы флуд не расходовал
  платные вызовы GigaChat API бесконтрольно.
- **Устойчивость к сбоям GigaChat API**: ретраи с backoff только для
  временных ошибок (лимит запросов, HTTP 5xx); ошибки авторизации
  (401), доступа (403) и некорректного запроса (400/404/413/422) не
  повторяются — это осознанные отказы, а не временный сбой.
- **Бот не выдумывает факты**: системный промпт явно требует
  отвечать только на основе контекста и прямо говорить о нехватке
  информации; при низком сходстве найденного контекста вопрос вообще
  не отправляется в LLM, а сразу эскалируется человеку — это и
  бизнес-требование, и снижение риска некорректных ответов клиентам.
- **Контейнер работает не от root** — в `Dockerfile` создан
  отдельный пользователь `appuser`.
- **Нет открытых входящих портов** — бот использует long polling, а
  не webhook, поэтому файрвол на сервере можно держать закрытым для
  всех входящих соединений, кроме SSH.
- **`GIGACHAT_VERIFY_SSL_CERTS=false` — только для локальной
  отладки.** Отключение проверки TLS-сертификата делает соединение с
  GigaChat уязвимым к атаке "человек посередине" (MITM). На
  продакшен-сервере обязательно используйте настоящий сертификат
  НУЦ Минцифры через `GIGACHAT_CA_BUNDLE_FILE` (см. DEPLOYMENT.md) и
  оставляйте `GIGACHAT_VERIFY_SSL_CERTS=true` (значение по
  умолчанию).

## Что нужно сделать при развёртывании (см. DEPLOYMENT.md)

- Вход на сервер только по SSH-ключу, пароль отключён.
- Файрвол (`ufw`) разрешает только SSH.
- Автоматические обновления безопасности системы включены.
- Реальные `BOT_TOKEN` и `GIGACHAT_CREDENTIALS` не публикуются
  нигде, включая скриншоты для портфолио — на скриншотах
  закрашивайте их вручную.

## Персональные данные клиентов

В базе хранятся вопросы клиентов, их chat_id, username и полное имя
(последние два — только в момент эскалации, в тексте уведомления
админу, не в отдельной таблице) — это минимально необходимый набор
для работы бота и аналитики `/stats`. Отдельного шифрования базы
данных или политики хранения (например, автоматического удаления
старых записей) в текущей версии не реализовано — при работе с
реальными клиентами реального бизнеса стоит обсудить с заказчиком
требования 152-ФЗ («О персональных данных») применительно к
конкретному случаю использования; это выходит за рамки технической
реализации данного проекта.

## Если токен или ключ API утекли

- **Telegram-токен**: немедленно отозвать через
  [@BotFather](https://t.me/BotFather) командой `/revoke` и получить
  новый.
- **GigaChat credentials**: отозвать и перевыпустить Authorization
  key в личном кабинете https://developers.sber.ru/studio/.

## Ограничения этого раздела

Этот проект — демонстрационный/учебный, не проходил профессиональный
аудит безопасности. Для реального коммерческого заказчика с большим
объёмом персональных данных клиентов потребуется отдельная, более
глубокая проработка (шифрование БД, политика хранения и удаления
данных, логи аудита действий админов и т.д.), которая не входит в
объём текущей реализации.
